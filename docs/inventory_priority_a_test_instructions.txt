Inventory Priority A Test Instructions (Updated for current codebase)
====================================================================

This guide explains how to test the **Inventory Priority A** sample SDFs by:
1) generating a **fresh ERP per SDF**, and
2) running a short manual checklist for each generated ERP.

Sample SDFs covered:
- test/sample_sdf_inventory_priority_a_reservations.json
- test/sample_sdf_inventory_priority_a_inbound.json
- test/sample_sdf_inventory_priority_a_cycle.json
- test/sample_sdf_inventory_priority_a_mixed.json


0) What each SDF is for
-----------------------

1. Reservations SDF
   - File: test/sample_sdf_inventory_priority_a_reservations.json
   - Purpose: Stock reservations + commit/release behavior, plus quantity integrity on the stock entity.

2. Inbound SDF
   - File: test/sample_sdf_inventory_priority_a_inbound.json
   - Purpose: Receiving workflow (PO -> GRN -> post/cancel) and stock updates.

3. Cycle SDF
   - File: test/sample_sdf_inventory_priority_a_cycle.json
   - Purpose: Cycle count lifecycle (start -> recalculate -> approve -> post) and variance posting.

4. Mixed SDF
   - File: test/sample_sdf_inventory_priority_a_mixed.json
   - Purpose: Run all Priority A packs together in one generated ERP and detect cross-pack regressions.


1) Prerequisites
----------------

- Run commands from repo root (PowerShell):
  - C:\Users\ASUS\CustomERP
- Node.js installed (use whatever version your repo currently uses for tests)
- If you want to start generated ERPs via Docker:
  - Docker Desktop installed and running

Note:
- These steps test the **generated ERP projects under generated/**.
- If you want to start the **platform** (not required for these instructions), see docs/local_testing_guide.md.


2) (Optional) Quick sanity check that generation works
------------------------------------------------------

From repo root:

node test/module_generation.integration.test.js

If this fails, fix generation first before spending time on manual inventory checks.


3) Generate an ERP from each sample SDF (one ERP per SDF)
---------------------------------------------------------

From repo root, run:

node test/run_assembler.js test/sample_sdf_inventory_priority_a_reservations.json
node test/run_assembler.js test/sample_sdf_inventory_priority_a_inbound.json
node test/run_assembler.js test/sample_sdf_inventory_priority_a_cycle.json
node test/run_assembler.js test/sample_sdf_inventory_priority_a_mixed.json

Each run prints:
- The generated project path under generated/<project-id>/
- A file/folder tree summary

IMPORTANT:
- Copy the printed "Generated project at: ..." path for each run so you can map:
  - which generated ERP came from which SDF.


4) Start a generated ERP (test one generated project at a time)
--------------------------------------------------------------

In PowerShell:

cd <generated-project-path>
.\dev.ps1 start

Then open:
- Backend API: http://localhost:3000
- Frontend UI: http://localhost:5173

Tip:
- Finish one SDF’s checklist, stop it, then start the next generated ERP.


5) Test checklist by SDF
------------------------

A) Reservations SDF checklist
-----------------------------

Goal: Verify reserve/commit/release and quantity integrity.

1) Create a product (example quantity: 100).
2) Reserve quantity 30.
   Expected:
   - reserved_quantity increases
   - available_quantity decreases (if available tracking is enabled by the generated config)
3) Commit the reservation.
   Expected:
   - reservation status becomes Committed
   - it cannot be committed/released again in an invalid state
4) Create a second reservation and then release it.
   Expected:
   - reserved_quantity decreases accordingly
5) Try invalid reserve inputs (negative, or more than allowed).
   Expected:
   - request rejected with validation/business error

API routes to verify (on the stock entity, usually products):
- GET  /api/products/:id/reservations
- POST /api/products/:id/reservations
- POST /api/products/:id/reservations/:reservationId/release
- POST /api/products/:id/reservations/:reservationId/commit


B) Inbound SDF checklist
------------------------

Goal: Verify PO/GRN receiving and stock updates.

1) Create product.
2) Create Purchase Order (PO).
3) Create PO line with ordered quantity.
4) Create Goods Receipt (GRN) linked to PO.
5) Create GRN lines linked to PO lines.
6) Post GRN.
   Expected:
   - product stock quantity increases
   - PO item received quantity updates
   - PO/PO item status becomes Partial or Received depending on totals
7) Cancel a draft GRN (not posted).
   Expected:
   - GRN status becomes Cancelled

API routes to verify:
- POST /api/goods_receipts/:id/post
- POST /api/goods_receipts/:id/cancel


C) Cycle SDF checklist
----------------------

Goal: Verify cycle counting lifecycle and variance posting.

1) Create product (example quantity: 100).
2) Create cycle count session.
3) Create a cycle count line referencing the product.
4) Start session.
   Expected: status -> InProgress
5) Set counted quantity on the line (example: 95).
6) Recalculate session.
   Expected:
   - expected_quantity is refreshed
   - variance_quantity is calculated (95 - 100 = -5)
   - session becomes PendingApproval when count data exists
7) Approve session.
   Expected: status -> Approved
8) Post session.
   Expected:
   - stock adjusts by variance (-5 in this example)
   - line/session statuses move to Posted

API routes to verify:
- POST /api/cycle_count_sessions/:id/start
- POST /api/cycle_count_sessions/:id/recalculate
- POST /api/cycle_count_sessions/:id/approve
- POST /api/cycle_count_sessions/:id/post


D) Mixed SDF checklist
----------------------

Goal: Verify the packs coexist and don’t regress each other.

1) Confirm the generated ERP exposes:
   - Reservations endpoints
   - Inbound (GRN post/cancel) endpoints
   - Cycle count (start/recalculate/approve/post) endpoints
2) Run one happy-path flow from each area.
3) Confirm no cross-pack breakage:
   - product quantities remain consistent
   - status transitions remain valid


6) Recommended test order
-------------------------

Run in this order:
1) reservations
2) inbound
3) cycle
4) mixed

This isolates pack-specific issues before the combined scenario.


7) Pass criteria (high level)
-----------------------------

A test run passes when:
- expected routes are present (no unexpected 404s)
- happy-path operations succeed
- invalid operations are blocked with clear errors
- stock quantities/statuses update correctly
- mixed scenario runs without cross-feature regression


8) Troubleshooting quick notes
------------------------------

- If generation fails:
  - Re-run the exact command and read assembler output (it will point to the failing mixin/pack/schema).
  - If multiple SDFs fail, run: node test/module_generation.integration.test.js
- If the generated ERP backend fails to start:
  - Use Docker start via .\dev.ps1 start, or start backend/frontend manually as printed by test/run_assembler.js.
- If workflow endpoints 404:
  - Confirm you started the ERP generated from the correct SDF (each SDF enables different packs).
  - Confirm the pack is enabled under modules.inventory in that SDF.
- If UI actions fail:
  - Check backend logs for validation/business-rule errors.
