# ENTITY MERGING RULES:

## 1. Shared Entities
When the same entity (by slug) appears in multiple module outputs:
- Merge their fields (union of all fields)
- Set `module: "shared"` 
- Preserve all features, ui, inventory_ops, etc. from both

Example: If Invoice has `customers` and Inventory also references `customers`:
- Create ONE `customers` entity with `module: "shared"`
- Include fields from both definitions

## 2. Module Assignment
- Entities from HR output: `module: "hr"` (unless shared)
- Entities from Invoice output: `module: "invoice"` (unless shared)
- Entities from Inventory output: `module: "inventory"` (unless shared)
- Entities used by 2+ modules: `module: "shared"`

## 3. Reference Validation
Ensure all `reference_entity` values point to valid entity slugs in the merged output.
If a reference is missing, add a warning.

## 4. Entity Order
Order entities logically:
1. Shared entities first (customers, products, etc.)
2. HR entities (departments, employees, leaves)
3. Invoice entities (invoices, invoice_items)
4. Inventory entities (products, locations, movements)

# FIELD MERGING RULES:
When merging fields from the same entity across modules:
- If same field name exists in both, prefer the more complete definition
- Keep all unique fields from both
- Preserve required, unique, options from the most restrictive definition

# CLARIFICATION MERGING:
- Combine all `clarifications_needed` from all modules
- Deduplicate by `id` (keep first occurrence)
- Ensure each question has: id, question, type, and options (if type is "choice")

# WARNINGS MERGING:
- Combine all warnings from all modules
- Add integration-specific warnings if needed (e.g., "employees entity is shared between HR and Inventory")

# SYSTEM FIELDS - DO NOT INCLUDE:
- `id`, `created_at`, `updated_at` - These are auto-generated

# DO NOT MODIFY:
- Do not invent new entities not present in the module outputs
- Do not remove entities from the module outputs
- Do not change field types unless fixing an obvious error
